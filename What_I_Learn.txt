WHAT I LEARNED
==============

[1] GPU COMPOSITING & COLOR POP
--------------------------------
When a CSS transform (like scale or translate) fires on an element, the browser
promotes that element to its own GPU compositor layer mid-animation. This
re-rasterization can cause a visible color flash or "glow pop" on nearby elements,
especially on saturated colors like red. This is a known Chromium issue called
the "GPU compositing color shift" or "layer promotion artifact."

Elements near the animated element (bystanders) can also flicker — when a neighbor
gets promoted to a GPU layer, surrounding elements get re-rasterized as a side effect.

FIX: "will-change: transform" pre-promotes an element to its own GPU layer before
any animation fires, preventing mid-animation re-rasterization. Apply it to the
animated element to prevent the pop on itself, or to sensitive bystanders to
insulate them from neighbor-triggered reshuffles.

COST: Every element with will-change: transform holds a permanent GPU texture,
consuming memory and battery even when idle. Avoid applying it to large lists.
For many elements, apply it dynamically on mouseenter and remove on mouseleave.

[2] STACKING TIMERS & CONCURRENT ASYNC EVENTS
----------------------------------------------
JavaScript is single-threaded — there are no real threads. But multiple
independent setTimeout timers can be running at the same time, and they
do not know about each other. This creates the same problems as multithreading:
race conditions, missed signals, and events firing in the wrong order.

In the battle system, two independent timers were racing each other:

  Timer A — Battle loop: fires BATTLE_STEP after 600ms
  Timer B — Animation clear: sets activeAnimations[id] = null after config.duration ms

Both start at the same moment when a step resolves. If an animation is longer
than 600ms (e.g. burn: 750ms, run_circle: 700ms), the battle loop fires first
and triggers a new React useEffect. That effect's cleanup function runs
`timers.forEach(clearTimeout)` — which cancels Timer B before it fires.
Result: the animation CSS class never gets cleared, and dead enemies never fade.

KEY INSIGHT: Timers stored in a local variable inside useEffect are owned by
that effect and get cancelled when the effect re-runs or unmounts. Timers stored
in a useRef are NOT cancelled by effect cleanup — they survive across re-renders
and step transitions. This is the difference between "effect-scoped" and
"persistent" timers.

FIX: Move timers that must always fire (like animation clear timers) from a
local array into a useRef. Store them the same way float timers already were:

  animClearTimersRef.current.push(setTimeout(() => {
    setActiveAnimations(prev => ({ ...prev, [id]: null }));
  }, config.duration));

Prune stale IDs from the ref at the start of each new step so they don't
accumulate across a long battle.

BROADER LESSON: When two async timers need to coordinate — one should drive
the other, not run independently. In this case, the battle loop delay was made
aware of animation durations (via battleDelay in the animation registry) so it
waits the right amount of time before advancing. This turns two racing timers
into one orchestrated sequence.

[3] SMALL ACCUMULATION BUGS THAT HIDE BEHIND USAGE PATTERNS
-------------------------------------------------------------
A tag system that merges stacks into a single growing value can produce wrong
results in ways that are very hard to notice — because the bug only surfaces
when the same mechanic is used more times than normal gameplay ever reaches.

In the SPEED_BOOST case, SpeedBoostOnApply merged each Speed Up into one tag
with a growing "amount" field. The stack count was derived with
Math.round(amount / tag.amount), which worked fine for 1 or 2 stacks.
The bug was invisible because in player gameplay Speed Up is only ever used
once per turn — never twice in a row. The math always happened to be correct
at that usage level.

Dark Fox Sumurai was the trigger: his AI uses Speed Up twice in the same turn,
which exposed the stale math. Stacks and amount got out of sync, producing
inflated speed numbers (e.g. 207 in the tooltip) that had no logical basis.

ROOT FIX: Each Speed Up application now pushes its own separate +20 entry into
the pool. SPEED_CALC sums them naturally (+20 per entry). IMBUE consumes each
one independently. No accumulation math, no derived stack count — the data
structure directly reflects reality.

BROADER LESSON: When designing a stacking system, prefer data that directly
represents state over data that is derived from other data. A list of separate
entries is easier to reason about than a single value with computed metadata.
And always test mechanics at 2-3x the normal usage depth — the player may only
ever use a card once, but an enemy AI might use it three times, and that is
where accumulation logic breaks.

RULE: Never accumulate a value that is read throughout the turn (like speed)
into a single growing tag. Speed is evaluated during SPEED_CALC for every
action — merging stacks into one amount means every subsequent read picks up
a value that was meant for only the next attack, corrupting the sort order of
unrelated actions.

Other values CAN be accumulated, but only if they are consumed entirely inside
a single action's execution bubble — damage multipliers, healing power, cost
modifiers. These are read once, applied once, and gone. They never bleed into
a separate action's calculation. Speed does not have this property, so it must
stay as separate independent entries that each action reads and discards on its
own terms.
