============================================================
  SLOT ORDER vs SPEED — HOW THEY WORK TOGETHER
============================================================

THE BATTLE LOOP
---------------
Every step of the battle loop re-evaluates ALL characters from
scratch. Each character submits their current queue[0] (front of
queue) to SpeedCheckAllAvailableActions, which sorts them all by
calc_speed descending. The single fastest action fires. That card
is removed from its owner's queue via ActionCleanup (queue.slice(1)).
Then the loop repeats.

This means the order is DYNAMIC — whatever happened in the previous
step changes who has what at queue[0] next step.

WHY PLAYER CARDS STILL FIRE IN SLOT ORDER (relative to each other)
--------------------------------------------------------------------
Each character only ever submits queue[0]. The player's slot 1 card
is at queue[1] and cannot compete until slot 0 has fired and been
removed, making slot 1 the new queue[0].

So player slot 0 ALWAYS fires before player slot 1, which ALWAYS
fires before slot 2 — relative to each other.

ENEMY INTERLEAVING
------------------
Enemy actions interleave dynamically between player cards based on
calc_speed each step. Example:

  Player queue: [slot0 calc_speed=50, slot1 calc_speed=80]
  Enemy queue:  [enemy0 calc_speed=70, enemy1 calc_speed=40]

  Step 1: Compare player queue[0]=50 vs enemy queue[0]=70
          → Enemy0 fires (70 wins). Enemy queue becomes [enemy1].

  Step 2: Compare player queue[0]=50 vs enemy queue[0]=40
          → Player slot0 fires (50 wins). Player queue becomes [slot1].

  Step 3: Compare player queue[0]=80 vs enemy queue[0]=40
          → Player slot1 fires (80 wins). Player queue becomes [].

  Step 4: Only enemy1 left → enemy1 fires.

Final order: enemy0 → slot0 → slot1 → enemy1

Note: slot1 having higher calc_speed (80) than slot0 (50) does NOT
let it jump ahead of slot0. It just affects how it interleaves with
enemies once it reaches queue[0].

RESOURCE LEGALITY RULE
-----------------------
For player resource planning, use SLOT INDEX — not calc_speed.
Player cards fire in slot order among themselves, so gains from
slot 0 are always available to slot 1 and slot 2.

  effectiveResourceAtExecution(type, mySlotIndex, queue, resources)
  → sums gains/costs from player slots 0..(mySlotIndex-1) only

DO NOT use calc_speed to determine resource availability between
player cards. Enemy actions don't affect player resources in the
current design, so interleaving with enemies is irrelevant here.

SPEED PENALTY APPLIES TO EVERYONE
----------------------------------
calcSpeed(base_speed, slot_index) = base_speed - (slot_index * 20)
This applies to BOTH player cards and enemy actions. Intentional.
Later actions in the queue are naturally slower.

SUMMARY
-------
  Battle loop      = dynamic re-evaluation every step (fastest wins)
  Player slot order = always respected (can only compete at queue[0])
  Speed penalty    = applies to both player and enemy, by slot/action index
  Resource planning = slot index only (not calc_speed)
============================================================
