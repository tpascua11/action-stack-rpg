PLAN: Battle Delay + Enemy Death Fade Fix
==========================================

## Problem 1 — Battle loop fires before animations finish

The battle loop setTimeout (600ms) and animation timers are independent.
Long animations (burn: 750ms, run_circle: 700ms) get cut off by the next step.

### Fix: Add optional `battleDelay` field to animationRegistry entries

Each animation in ANIMATIONS can define:
  - duration     — how long the visual CSS animation plays (already exists)
  - battleDelay  — how long the battle loop should wait before firing the next step
                   (optional — falls back to `duration` if not set)

This separates two concerns:
  - Visual timing  (duration)
  - Game pacing    (battleDelay)

Example registry entry:
  big_spell: {
    cssClass: 'animate-big_spell',
    duration: 1200,       // visual plays for 1200ms
    battleDelay: 900,     // loop waits 900ms (tail is just fade, not critical)
    sfx: SFX_SOMETHING,
    floatingNumber: { color: '#a78bfa' },
  }

If battleDelay is omitted, duration is used as the delay.
This also future-proofs spritesheet animations — duration = spritesheet
playback length, battleDelay = independent game-timing knob.

### Changes needed:

1. animationRegistry.js
   - No required changes to existing entries (battleDelay is opt-in)
   - Add battleDelay to any animations where you want pacing different from duration

2. BattleScreen.jsx — battle loop useEffect
   Replace hardcoded 600ms with a computed delay:

   const maxDelay = (gs.pendingAnimation ?? []).reduce((max, anim) => {
     const config = ANIMATIONS[anim.type];
     if (!config) return max;
     return Math.max(max, config.battleDelay ?? config.duration ?? 0);
   }, 0);
   const delay = Math.max(600, maxDelay + 150); // 150ms buffer

   battleTimerRef.current = setTimeout(() => {
     dispatch({ type: 'BATTLE_STEP' });
   }, delay);

## Problem 2 — Enemy death fade starts too early

When an enemy's health hits 0, EnemyZone immediately applies opacity-30
via a CSS transition (transition-all duration-300). But the killing blow
animation (e.g. shake: 350ms) is still playing at the same moment.
Result: the enemy shakes and fades simultaneously, which looks wrong.
The enemy should finish receiving the hit before it fades out.

### Current behavior (EnemyZone.jsx line 67):
  className={`... ${isDead ? 'opacity-30' : 'opacity-100'} ...`}
  — opacity change is immediate/parallel with the hit animation

### Fix: Delay the fade until the killing animation finishes

Track "pending death fade" in EnemyZone local state.
When isDead flips to true and there's an active animation on that enemy,
wait for the animation to finish before applying the low opacity.

Approach — use a local `dyingEnemies` Set in EnemyZone:
  - When an enemy's isDead becomes true for the first time, add it to dyingEnemies
  - After (animDuration + 150ms) remove it from dyingEnemies
  - Render: use opacity-30 only if isDead AND NOT in dyingEnemies

This means the enemy plays its death-blow animation at full opacity,
then fades out cleanly after.

### Changes needed:

1. EnemyZone.jsx
   - Add local state: const [dyingEnemies, setDyingEnemies] = useState(new Set())
   - useEffect watching enemies + activeAnimations:
       enemies.forEach(enemy => {
         if (enemy.health <= 0 && !dyingEnemies.has(enemy.id)) {
           const anim = activeAnimations[enemy.id];
           const config = anim ? ANIMATIONS[anim.cssClass reverse-lookup] : null;
           const delay = config ? config.duration + 150 : 150;
           setTimeout(() => {
             setDyingEnemies(prev => new Set([...prev, enemy.id]));
           }, delay);
         }
       });
   - Render condition:
       const isFaded = isDead && dyingEnemies.has(enemy.id);
       className={`... ${isFaded ? 'opacity-30' : 'opacity-100'} ...`}

   NOTE: dyingEnemies should be cleared on unmount / new battle to avoid stale state.

   SIMPLER ALTERNATIVE: pass activeAnimations into EnemyZone (already done),
   and use inline transitionDelay on the opacity:
       style={{ transitionDelay: isDead && anim ? `${animDuration}ms` : '0ms' }}
   This avoids extra state but requires a duration lookup from the registry.

## Order of implementation

1. animationRegistry.js — add battleDelay opt-in field (no breaking changes)
2. BattleScreen.jsx — update battle loop to use battleDelay/duration
3. EnemyZone.jsx — delay death fade until killing animation finishes

---

## Notes for future spritesheet support

When spritesheets land:
  - duration = spritesheet total frame time
  - battleDelay = separate knob, unaffected
  - No changes needed to the battle loop timing logic
